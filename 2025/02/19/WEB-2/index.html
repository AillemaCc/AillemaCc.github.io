应用程序上下文和输出目录的关系

可能有点抽象，我们再解释一下这个内容

应用程序上下文（Context Path）

  • 定义：应用程序上下文是指你的 Web 应用在 Tomcat 服务器上的虚拟路径前缀。它用于区分在同一 Tomcat 实例上运行的不同 Web 应用。
  • 作用：它决定了用户通过浏览器访问应用时的 URL 路径部分。例如，如果你的应用程序上下文设置为 /myapp，那么用户可以通过 http://localhost:8080/myapp 来访问该应用。

工件的输出目录

  • 定义：工件的输出目录是构建工具（如 IntelliJ IDEA 或 Maven）生成并存放编译后的文件、资源和依赖库的地方。对于 exploded 工件，这个目录是一个展开的、未压缩的结构，包含了所有需要部署到服务器的文件。
  • 作用：这个目录包含了你的应用的所有必要组件，包括 Java 类文件、静态资源（HTML、CSS、JavaScript 文件等）、配置文件以及依赖库。Tomcat 会从这个目录中读取这些文件来提供服务

虽然应用程序上下文和工件的输出目录是两个不同的概念，但它们在部署过程中是相关的：

  1. 映射关系：
    • 当你将一个 exploded 工件部署到 Tomcat 时，你需要指定应用程序上下文。Tomcat 使用这个上下文路径来决定如何路由请求到你的应用。
    • 例如，如果你的应用程序上下文是 /bookstore，当用户访问 http://localhost:8080/bookstore 时，Tomcat 会将请求路由到你的 exploded 工件目录中的相应位置。
  2. 部署配置：
    • 在 IntelliJ IDEA 中，当你配置一个 exploded 工件进行部署时，IDEA 会在其内部创建一个临时的工作目录，并将你的工件内容映射到这个工作目录下的适当位置。然后，Tomcat 会使用这个工作目录来提供服务。
    • 具体来说，IDEA 会根据你在 Run/Debug Configurations -> Deployment 中指定的应用程序上下文来设置映射关系。
  • 应用程序上下文 是用来定义你的应用在 Tomcat 上的虚拟路径，帮助定位和隔离不同的应用。
  • 工件的输出目录 是包含所有应用文件和资源的实际物理位置，Tomcat 从这里读取文件来提供服务。
  • 虽然应用程序上下文本身并不直接“定位”工件的输出目录，但它与输出目录一起工作，确保正确的请求被路由到正确的应用资源。换句话说，应用程序上下文告诉 Tomcat 如何处理请求，而工件的输出目录提供了实际的文件内容。

而且应用程序上下文的修改和url的修改是同步的

ServletContent的使用

ServletContext 是 Java Servlet API 中的一个核心接口，它提供了访问整个 Web 应用的上下文信息和资源的功能。ServletContext 对象代表了整个 Web 应用程序的环境，并且在应用的生命周期内是唯一的。每个部署的 Web 应用都有一个与之关联的 ServletContext 实例。

主要功能：

共享数据：

ServletContext 提供了一个全局作用域，可以在整个 Web 应用中共享数据。你可以使用 setAttribute、getAttribute 和 removeAttribute 方法来设置、获取和移除属性。

访问初始化参数：

可以通过 ServletContext 访问在 web.xml 文件或注解中定义的应用级初始化参数（区别于 ServletConfig 提供的 Servlet 级初始化参数）

获取资源路径：

ServletContext 提供了获取应用中资源路径的方法，如 getResourceAsStream 和 getResourcePaths，可以用来读取配置文件、静态资源等。

日志记录：

可以通过 log 方法将消息写入到应用的日志文件中。

转发请求事实上是一个关键语句：

这行代码用于在Java Servlet环境中实现请求转发（Request Dispatch）。让我们详细解析一下这个过程：

请求转发的工作原理

  1. 获取分发器（Dispatcher）:
    • request.getRequestDispatcher("/SecondServlet")：这行代码从当前的HttpServletRequest对象中获取一个RequestDispatcher对象。"/SecondServlet"是你要转发到的目标资源的路径。这个路径可以是一个相对路径或绝对路径，取决于你的应用结构。
  2. 执行转发:
    • .forward(request, response)：通过调用RequestDispatcher对象的forward方法，将当前请求和响应转发给指定的资源（在这个例子中是/SecondServlet）。这意味着原始请求的所有数据（包括参数、属性等）都会被传递给目标Servlet。

转发的具体步骤

  • 保持请求和响应对象：在转发过程中，原始的HttpServletRequest和HttpServletResponse对象会被保持并传递给目标Servlet。这意味着目标Servlet可以访问原始请求中的所有数据。
  • 控制权转移：一旦调用了forward方法，当前Servlet的处理就会停止，并且控制权会转移到目标Servlet。如果当前Servlet中有任何后续代码，在调用forward之后都不会被执行。
  • 输出流管理：需要注意的是，如果你已经在当前Servlet中向HttpServletResponse对象写入了内容（例如，使用response.getWriter().println("Some content");），那么尝试进行转发将会导致IllegalStateException异常。这是因为HTTP响应只能有一个主体，一旦开始写入就不能再转发请求。
