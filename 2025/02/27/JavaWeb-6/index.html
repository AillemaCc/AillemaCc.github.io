INNER JOIN 是SQL（结构化查询语言）中用于合并两个或多个表的数据的一种连接类型。它基于相关列的值匹配行，仅返回那些在被连接的表中都有对应匹配的记录。换句话说，只有当连接条件满足时，INNER JOIN 才会返回结果集中的行。

基本概念

  • 连接条件：通常通过一个共同的字段来指定，这个字段在两个表中都存在，并且代表相同的数据。
  • 结果集：只包含那些在两个表中都能找到匹配记录的行。如果某个表中的某条记录在另一个表中没有匹配项，则这条记录不会出现在结果集中。

示例

  1. Customers 表，包含客户信息，如 CustomerID, CustomerName
  2. Orders 表，包含订单信息，如 OrderID, CustomerID, OrderDate

关键点

  • 匹配性：只有两边都有匹配记录时才会显示结果。
  • 数据完整性：有助于确保查询结果反映了数据库中数据的真实关系。

INNER JOIN 是最常用的连接类型之一，适用于需要根据某些条件组合两个表中的数据的情况。通过明确指定连接条件，可以使查询更加精确和有效。

外连接

外连接分为两种：左外连接 和 右外连接。

左外连接语法结构：

左外连接相当于查询表1(左表)的所有数据，当然也包含表1和表2交集部分的数据。

右外连接语法结构：

右外连接相当于查询表2(右表)的所有数据，当然也包含表1和表2交集部分的数据。

在SQL查询中，ON 子句和 WHERE 子句有不同的作用和用途。理解它们的区别有助于编写更准确、高效的查询。

ON 子句 vs WHERE 子句

  • ON 子句：用于指定连接条件，即决定如何将两个表中的记录进行匹配。它定义了哪些行应该被组合在一起。
  • WHERE 子句：用于过滤最终的结果集，即在所有表的记录已经通过 ON 条件连接之后，进一步筛选出满足特定条件的行。

为什么需要分开？

  1. 连接条件（ON 子句）：
    • 定义了两个表之间如何关联。
    • 例如，在你的查询中，d.category_id = c.id 是用来确定 dish 表中的每一条记录与 category 表中的哪一条记录相关联。
  2. 过滤条件（WHERE 子句）：
    • 在连接操作完成之后，对生成的结果集进行进一步过滤。
    • 例如，在你的查询中，d.price BETWEEN 10 AND 50 AND d.status = 1 是用来筛选出符合条件的菜品记录。

能否把所有条件都写在 ON 子句中？

理论上，你可以将一些条件放在 ON 子句中，但这会导致不同的结果，特别是在使用外连接（如左外连接 LEFT JOIN）时。具体来说：

  • 连接条件：必须放在 ON 子句中，因为它们决定了如何匹配两个表的记录。
  • 过滤条件：通常应放在 WHERE 子句中，以确保在连接操作完成后对结果集进行过滤。

如果将过滤条件放在 ON 子句中，尤其是在外连接的情况下，可能会导致不正确的结果。例如：

子查询

介绍

SQL语句中嵌套select语句，称为嵌套查询，又称子查询。

子查询外部的语句可以是insert / update / delete / select 的任何一个，最常见的是 select。

根据子查询结果的不同分为：

  1. 标量子查询（子查询结果为单个值[一行一列]）

  2. 列子查询（子查询结果为一列，但可以是多行）

  3. 行子查询（子查询结果为一行，但可以是多列）

  4. 表子查询（子查询结果为多行多列[相当于子查询结果是一张表]）

子查询可以书写的位置：

  1. where之后
  2. from之后
  3. select之后

标量子查询

子查询返回的结果是单个值(数字、字符串、日期等)，最简单的形式，这种子查询称为标量子查询。

常用的操作符： = <> > >= < <=

列子查询

子查询返回的结果是一列(可以是多行)，这种子查询称为列子查询。

常用的操作符：

事务

场景：学工部整个部门解散了，该部门及部门下的员工都需要删除了。

  • 操作：

  • 问题：如果删除部门成功了，而删除该部门的员工时失败了，此时就造成了数据的不一致。

​ 要解决上述的问题，就需要通过数据库中的事务来解决。

介绍

在实际的业务开发中，有些业务操作要多次访问数据库。一个业务要发送多条SQL语句给数据库执行。需要将多次访问数据库的操作视为一个整体来执行，要么所有的SQL语句全部执行成功。如果其中有一条SQL语句失败，就进行事务的回滚，所有的SQL语句全部执行失败。

简而言之：事务是一组操作的集合，它是一个不可分割的工作单位。事务会把所有的操作作为一个整体一起向系统提交或撤销操作请求，即这些操作要么同时成功，要么同时失败。

事务作用：保证在一个事务中多次操作数据库表中数据时，要么全都成功,要么全都失败。

操作

MYSQL中有两种方式进行事务的操作：

  1. 自动提交事务：即执行一条sql语句提交一次事务。（默认MySQL的事务是自动提交）
  2. 手动提交事务：先开启，再提交

事务操作有关的SQL语句：

SQL语句 描述
start transaction; / begin ; 开启手动控制事务
commit; 提交事务
rollback; 回滚事务

手动提交事务使用步骤：

  • 第1种情况：开启事务 => 执行SQL语句 => 成功 => 提交事务
  • 第2种情况：开启事务 => 执行SQL语句 => 失败 => 回滚事务

使用事务控制删除部门和删除该部门下的员工的操作：

  • 上述的这组SQL语句，如果如果执行成功，则提交事务
  • 上述的这组SQL语句，如果如果执行失败，则回滚事务

四大特性

面试题：事务有哪些特性？

  • 原子性（Atomicity）：事务是不可分割的最小单元，要么全部成功，要么全部失败。
  • 一致性（Consistency）：事务完成时，必须使所有的数据都保持一致状态。
  • 隔离性（Isolation）：数据库系统提供的隔离机制，保证事务在不受外部并发操作影响的独立环境下运行。
  • 持久性（Durability）：事务一旦提交或回滚，它对数据库中的数据的改变就是永久的。

事务的四大特性简称为：ACID

  • 原子性（Atomicity） ：原子性是指事务包装的一组sql是一个不可分割的工作单元，事务中的操作要么全部成功，要么全部失败。

  • 一致性（Consistency）：一个事务完成之后数据都必须处于一致性状态。

​ 如果事务成功的完成，那么数据库的所有变化将生效。

​ 如果事务执行出现错误，那么数据库的所有变化将会被回滚(撤销)，返回到原始状态。

  • 隔离性（Isolation）：多个用户并发的访问数据库时，一个用户的事务不能被其他用户的事务干扰，多个并发的事务之间要相互隔离。

​ 一个事务的成功或者失败对于其他的事务是没有影响。

  • 持久性（Durability）：一个事务一旦被提交或回滚，它对数据库的改变将是永久性的，哪怕数据库发生异常，重启之后数据亦然存在。

索引

介绍

索引(index)：是帮助数据库高效获取数据的数据结构 。

  • 简单来讲，就是使用索引可以提高查询的效率。

优点：

  1. 提高数据查询的效率，降低数据库的IO成本。
  2. 通过索引列对数据进行排序，降低数据排序的成本，降低CPU消耗。

缺点：

  1. 索引会占用存储空间。
  2. 索引大大提高了查询效率，同时却也降低了insert、update、delete的效率。

8. 其他注意事项

  • 索引的选择性：高选择性的列（即列中有大量不同的值）适合创建索引。
  • 索引的代价：虽然索引可以加快查询速度，但它们也会增加插入、更新和删除操作的成本，因为每次这些操作发生时，索引也需要更新。
  • 覆盖索引：如果查询的所有列都在索引中，数据库可以直接从索引中获取结果，而不需要访问实际的数据行，这称为覆盖索引。

总结

索引是数据库性能优化的重要工具。常见的索引操作包括：

  • 创建索引：使用 CREATE INDEX 或 CREATE UNIQUE INDEX。
  • 查看索引：使用 SHOW INDEX 或查询系统视图。
  • 删除索引：使用 DROP INDEX。
  • 维护索引：定期重建索引和更新统计信息。

根据具体的数据库管理系统（如MySQL、PostgreSQL、SQL Server等），具体的语法可能会有所不同，但基本概念和操作是相似的。正确使用索引可以显著提高数据库查询的效率。
