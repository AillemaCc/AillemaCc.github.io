生成HTTP请求消息

URL

URL（统一资源定位符）是因特网中的唯一资源的地址。它是浏览器用于检索已发布资源（例如 HTML 页面、CSS 文档、图像等）的关键机制之一。

任何网址都是URL，比如说：

你可以看到这三个url的长度不尽相同，分割的符号将他们分成不同的部分

URL 确实由不同的部分组成，其中一些是必须的，而另一些是可选的。最重要的部分以在下面的 URL 上高亮（详细信息在下面的各节中提供）：

URL-Scheme

URL 的第一部分是Scheme，表示浏览器必须使用的协议来请求资源。协议是计算机网络当中交换或者传输数据的一组方法。通常对于网站来说，协议是 HTTPS 或 HTTP。访问网页需要这两者之一，但浏览器还知道如何处理其他方案，比如 mailto:（打开邮件客户端），所以如果你看到其他协议也不要感到惊讶。

URL-Authority

接下来是权威（authority），它与方案之间用字符模式 :// 分隔。如果存在，权威会包括域（例如 www.example.com）和端口（80），由冒号分隔：

但我们一般单独介绍域名(Domain Name)和端口(Port)

域指示被请求的 Web 服务器。通常这是一个域名，但也可以使用 IP 地址（但这很少见，因为它不太方便）。

端口指示用于访问 Web 服务器上资源的技术“门户”。如果 Web 服务器使用 HTTP 协议的标准端口（HTTP 为 80，HTTPS 为 443）来授予对其资源的访问权限，则通常会省略端口。否则，端口是强制的

URL-资源路径

/path/to/myfile.html 是 Web 服务器上资源的路径。在 Web 的早期阶段，像这样的路径表示 Web 服务器上的物理文件位置。如今，它主要是由没有任何物理现实的 Web 服务器处理的抽象路径。

URL-参数

?key1=value1&key2=value2 是提供给 Web 服务器的额外参数。这些参数是用 & 符号分隔的键/值对列表。在返回资源之前，Web 服务器可以使用这些参数来执行额外的操作。每个 Web 服务器都有自己关于参数的规则

URL-解析

浏览器要做的第一步工作就是对 URL 进行解析，从而生成发送给 Web服务器的请求消息。当解析发生时，需要按照上面的形式对URL进行拆分，拆分的结果能让浏览器明白URL的含义。比如说www.lab.glassroom.com/dirl/filel.html就代表着我们要访问
www.lab.glasscom.com 这 个 Web 服 务 器 上 路 径 名 为 /dir/file1.html 的 文件，也就是位于 /dir/ 目录 D 下的 file1.html 这个文件

也会有省略文件名的情况出现，比如直接访问：

http://www.lab.glassroom.com/dir/

这时候可以这样理解，以“/”结尾代表 /dir/ 后面本来应该有的文件名被省略了。根据 URL 的规则，文件名可以像前面这样省略。省略了文件名，我们就会提前设置好要默认访问的文件名。这个设置因人而异，大多数时候是index.html之类的。

或者这样：

http://www.lab.glasscom.com/

也可以默认访问

或者这样：

http://www.lab.glasscom.com

那么我们就可以配置默认文件为/index.html这样url一拼接，也可以执行默认访问

HTTP的基本思路

解析完成url之后，我们就知道要访问的目标在哪里。接下来，浏览器就会使用HTTP协议来访问web服务器

该协议定义了客户端和服务器之间交互的消息内容和步骤，其基本思路为：客户端向服务器发送请求消息，请求消息当中包含的内容是“对什么(target)”和”进行怎样的操作”两部分。其中相当于target的部分称为URL，URL我们已经介绍过。

进行怎样的操作 这一部分称为方法。方法表示需要让web服务器完成怎样的工作。其中典型的例子包括读取 URI 表示的数据、将客户端输入的数据发送给 URI 表示的程序等。下面这张表列举了主要的方法

除了上面的内容之外，HTTP消息当中还有一些用来表示附加消息的头字段。客户端向web服务器发送数据时，会先发送头字段，会先发送头字段，然后再发送数据。不过，头字段属于可有可无的附加信息，因此我们留到后面再讲。

收到请求消息之后，web服务器会请求消息的内容进行解析，通过URI和方法来判断target和operation，并且根据这些要求来完成自己的工作，然后将结果存放在响应消息当中。响应消息就是一种反馈，告诉你你的请求产生了什么结果

在响应消息中，有一个状态码它用来表示操作的执行结果是成功还是发生了错误。比如404.其实这就是状态码。状态码后面就是头字段和网页数据。响应消息会被发送回客
户端，客户端收到之后，浏览器会从消息中读出所需的数据并显示在屏幕上。到这里，HTTP 的整个工作就完成了。

HTTP方法–GET&POST

这两个方法都很常用，我们以这两个方法来介绍HTTP方法的基本思路

一般当我们访问 Web 服务器获取网页数据时，使用的就是 GET 方法。所谓一般的过程大概就是这样的：首先，在请求消息中写上 GET 方法，然后在 URI 中写上存放网页数据的文件名“/dir1/file1.html”，这就表示我们需要获取 /dir1/file1.html 文件中的数据。当 Web 服务器收到消息后，会打开 /dir1/file1.html 文件并读取出里面的数据，然后将读出的数据存放到响应消息中，并返回给客户端。最后，客户端浏览器会收到这些数据并显示在屏幕上。

还有一个经常使用的方法就是 POST，我们在表单中填写数据并将其发送给 Web 服务器时就会使用这个方法。当我们在网上商城填写收货地址和姓名，或者是在网上填写问卷时，都会遇到带有输入框的网页，而这些可以输入信息的部分就是表单。使用 POST 方法时，URI 会指向 Web 服务器中运行的一个应用程序 B 的文件名，典型的例子包括“index.cgi”“index.php”等。然后，在请求消息中，除了方法和 URI 之外，还要加上传递给应用程序和脚本的数据。这里的数据也就是用户在输入框里填写的信息。当服务器收到消息后，Web 服务器会将请求消息中的数据发送给 URI 指定的应用程序。最后，Web 服务器从应用程序接收输出的结果，会将它存放到响应消息中并返回给客户端

生成HTTP请求信息

重新回到浏览器这个视角，当对url进行解析之后，浏览器确定了web服务器和文件名，接下来就是根据这些信息来确定HTTP请求消息。事实上，HTTP消息在格式上存在着严格规定，因此浏览器会按照规定的格式来生成请求消息

首先，请求消息的第一行称为请求行，请求行的重点是其最开头的方法。方法可以告诉目标的web服务器，告诉他应该进行什么样的操作。不过在这里，我们需要解决一个问题，就是方法是多种多样的，我们必须要先判断，应该选用其中的哪一种方法

选用什么方法当然是根据需求，比如我们需要在地址栏当中输入网址，显示网页，那我们就GET。如果我们需要修改文章，那就POST

第一行的末尾要写上HTTP的版本号，这是为了表示该消息基于那个版本的HTTP规格编写的。中间的URI一般是文件和程序的路径名

这样，第一行就结束了

第二行开始是消息头，尽管通过第一行，我们可以大致理解请求的内容，但部分情况下，还需要额外的信息，消息头的功能就是存放这些信息。消息头的规格中定义了很多项目，如日期、客户端支持的数据类型、语言、压缩格式、客户端和服务器的软件名称和版本、数据有效期和最后更新时间等。

写完消息头之后，还需要添加一个完全没有内容的空行，然后写上需要发送的数据。这一部分称为消息体，也就是消息的主体。不过，在使用GET 方法的情况下，仅凭方法和 URI，Web 服务器就能够判断需要进行怎样的操作，因此消息体中不需要填写任何数据。消息体结束之后，整个消息也就结束了。
当使用 POST 方法时，需要将表单中填写的信息写在消息体中。到此
为止，请求消息的生成操作就全部完成了。

发送请求–>收到响应

当我们将上述请求消息发送出去之后，Web 服务器会返回响应消息。响应消息的格式以及基本思路和请求消息是相同的，差别只在第一行上。

在响应消息中，第一行的内容为状态码和响应短语，用来表示请求的执行结果是成功还是出错。状态码和响应短语表示的内容一致，但它们的用途不同。状态码是一个数字，它主要用来向程序告知执行的结果，相对地，响应短语则是一段文字，用来向人们告知执行的结果。

返回响应消息之后，浏览器会将数据提取出来并显示在屏幕上，我们就能够看到网页的样子了。如果网页的内容只有文字，那么到这里就全部处理完毕了，但如果网页中还包括图片等资源，则还有下文。

当网页中包含图片时，会在网页中的相应位置嵌入表示图片文件的标签的控制信息。浏览器会在显示文字时搜索相应的标签，当遇到图片相关的标签时，会在屏幕上留出用来显示图片的空间，然后再次访问 Web 服务器，按照标签中指定的文件名向 Web 服务器请求获取相应的图片并显示在预留的空间中。这个步骤和获取网页文件时一样，只要在 URI 部分写上图片的文件名并生成和发送请求消息就可以了

每条请求消息中只能写 1 个 URI，所以每次只能获取 1 个文件，如果需要获取多个文件，必须对每个文件单独发送 1 条请求。比如 1 个网页中包含 3 张图片，那么获取网页加上获取图片，一共需要向 Web 服务器发送 4 条请求。
