如何生成分布式雪花算法ID

首先，分布式系统中，有一些需要使用全局唯一 ID 的场景，这种时候为了防止 ID 冲突可以使用 36 位的 UUID，但是 UUID 有一些缺点，首先他相对比较长，另外 UUID 一般是无序的。

有些时候我们希望能使用一种简单些的 ID，并且希望 ID 能够按照时间有序生成。

什么是雪花算法

Snowflake 中文的意思是雪花，所以常被称为雪花算法，是 Twitter 开源的分布式 ID 生成算法。

Twitter 雪花算法生成后是一个 64bit 的 long 型的数值，组成部分引入了时间戳，基本保持了自增。

SnowFlake 算法的优点：

  1. 高性能高可用：生成时不依赖于数据库，完全在内存中生成。
  2. 高吞吐：每秒钟能生成数百万的自增 ID。
  3. ID 自增：存入数据库中，索引效率高。

SnowFlake 算法的缺点：

依赖与系统时间的一致性，如果系统时间被回调，或者改变，可能会造成 ID 冲突或者重复。

1. 雪花算法组成

snowflake 结构如下图所示：

包含四个组成部分：

不使用：1bit，最高位是符号位，0 表示正，1 表示负，固定为 0。

时间戳：41bit，毫秒级的时间戳（41 位的长度可以使用 69 年）。

标识位：5bit 数据中心 ID，5bit 工作机器 ID，两个标识位组合起来最多可以支持部署 1024 个节点

序列号：12bit 递增序列号，表示节点毫秒内生成重复，通过序列号表示唯一，12bit 每毫秒可产生 4096 个 ID。

通过序列号 1 毫秒可以产生 4096 个不重复 ID，则 1 秒可以生成 4096 * 1000 = 409w ID。

默认的雪花算法是 64 bit，具体的长度可以自行配置。如果希望运行更久，增加时间戳的位数；如果需要支持更多节点部署，增加标识位长度；如果并发很高，增加序列号位数。

总结：雪花算法并不是一成不变的，可以根据系统内具体场景进行定制。

2. 雪花算法适用场景

因为雪花算法有序自增，保障了 MySQL 中 B+ Tree 索引结构插入高性能。

所以，日常业务使用中，雪花算法更多是被应用在数据库的主键 ID 和业务关联主键。

雪花算法生成 ID 重复问题

假设：一个订单微服务，通过雪花算法生成 ID，共部署三个节点，标识位一致。 此时有 200 并发，均匀散布三个节点，三个节点同一毫秒同一序列号下生成 ID，那么就会产生重复 ID。

它们的ID生成逻辑完全一致：

  • 时间戳：同一毫秒内相同。
  • 机器标识：完全相同。
  • 序列号：从0开始自增。

此时，三个节点在同一毫秒内生成的ID仅依赖序列号，而序列号是独立维护的（每个节点有自己的序列号计数器）。如果并发请求导致：

  • 三个节点在同一毫秒内生成ID。
  • 三个节点的序列号计数器恰好同时从0开始（如刚启动或序列号耗尽归零）。

则会生成完全相同的ID。

前提条件

  1. 服务通过集群的方式部署，其中部分机器标识位一致。
  2. 业务存在一定的并发量，没有并发量无法触发重复问题。
  3. 生成 ID 的时机：同一毫秒下的序列号一致。

1. 标识位如何定义

如果能保证标识位不重复，那么雪花 ID 也不会重复。

通过上面的案例，知道了 ID 重复的必要条件。如果要避免服务内产生重复的 ID，那么就需要从标识位上动文章。

我们先看看开源框架中使用雪花算法，如何定义标识位。

Mybatis-Plus v3.4.2 雪花算法实现类 Sequence，提供了两种构造方法：无参构造，自动生成 dataCenterId 和 workerId；有参构造，创建 Sequence 时明确指定标识位。

Hutool v5.7.9 参照了 Mybatis-Plus dataCenterId 和 workerId 生成方案，提供了默认实现

一起看下 Sequence 的创建默认无参构造，如何生成 dataCenterId 和 workerId。

dataCenterId

基于机器MAC地址生成数据中心ID（DataCenter ID）

这段代码的作用是基于机器MAC地址生成数据中心ID（DataCenter ID），通常用于分布式ID生成算法（如雪花算法Snowflake）中，确保不同机器的数据中心ID唯一。以下是逐步解析：

代码解析

关键步骤说明

1. 获取MAC地址

  • NetUtil.getLocalHardwareAddress() 返回本机的MAC地址（6字节的byte[]，如 [0x00, 0x1A, 0x2B, 0x3C, 0x4D, 0x5E]）。

2. 提取最后两个字节

  • mac[mac.length - 2]：倒数第二个字节（如 0x4D）。
  • mac[mac.length - 1]：倒数第一个字节（如 0x5E）。
  • 将这两个字节合并为一个16位整数：

3. 右移6位（相当于除以64）

  • 0x5E4D >> 6 = 0x179（即 0x5E4D / 64 = 241）。
  • 目的：将16位的MAC地址片段压缩到更小的范围（避免后续取模后冲突概率过高）。

4. 取模运算限制范围

  • id % (maxDatacenterId + 1)：确保生成的ID不超过允许的最大值（如 maxDatacenterId=31，则结果范围是 0~31）。

5. 默认值

  • 如果MAC地址获取失败（mac=null），返回默认值 1。

为什么用MAC地址？

  1. 唯一性：MAC地址通常是全局唯一的（除非手动篡改）。
  2. 稳定性：服务器物理网卡一般不会频繁变更，适合作为机器标识的生成依据。

示例计算

假设：

  • MAC地址：[0x00, 0x1A, 0x2B, 0x3C, 0x4D, 0x5E]（最后两字节 0x4D, 0x5E）。
  • maxDatacenterId = 31。

计算过程：

  1. 合并最后两字节：
    0x004D | (0x005E << 8) = 0x5E4D（十进制 24141）。
  2. 右移6位：
    0x5E4D >> 6 = 0x179（即 24141 / 64 = 241）。
  3. 取模：
    241 % 32 = 17（因为 maxDatacenterId + 1 = 32）。

最终返回 17。

潜在问题

  1. MAC地址冲突：

    • 虚拟化环境中（如Docker/K8s），MAC地址可能重复。
    • 解决方案：结合其他信息（如IP地址）生成唯一ID。

  2. 物理网卡不可用：

    • 云服务器可能禁用MAC地址访问。
    • 解决方案：回退到其他唯一标识（如主机名哈希）。

  3. 取模后的冲突：

    • 不同MAC地址计算后可能映射到相同的dataCenterId。
    • 优化：增加哈希函数分散度。

改进建议

总结

  • 核心逻辑：通过MAC地址的最后两个字节生成唯一ID，并限制在指定范围内。
  • 适用场景：分布式系统中需要稳定、唯一的机器标识时（如雪花算法的datacenterId）。
  • 注意事项：需确保MAC地址的唯一性和可访问性，否则需备选方案。

workid

方法功能

这段代码的作用是基于数据中心ID（datacenterId）和进程ID（PID）生成工作节点ID（workerId），通常用于雪花算法（Snowflake）中，确保同一数据中心内的不同工作节点ID唯一。

代码解析

关键步骤原理

1. 拼接唯一标识字符串

  • datacenterId：传入的数据中心ID（确保不同数据中心的workerId不冲突）。
  • RuntimeUtil.getPid()：获取当前JVM进程的PID（确保同一数据中心内不同进程的workerId不冲突）。
  • 最终拼接字符串示例："31742"（假设 datacenterId=3，PID=1742）。

2. 生成哈希值

  • mpid.toString().hashCode()：
    对拼接后的字符串调用Java默认的哈希算法（返回int类型）。
    例如 "31742".hashCode() 可能返回 123456789。
  • & 0xffff：
    取哈希值的低16位（相当于 hashCode % 65536），将结果限制在 0~65535 范围内。
    例如 123456789 & 0xffff = 0x8EB5（十进制 36533）。

3. 取模限制范围

  • % (maxWorkerId + 1)：
    将哈希值映射到 [0, maxWorkerId] 范围内。
    例如若 maxWorkerId=31，则 36533 % 32 = 21。

4. 异常处理

  • 如果获取PID失败（如无权限或非Linux系统），仅使用 datacenterId 生成哈希值，冲突概率会升高。

为什么这样设计？

  1. 唯一性保障：

    • 同一数据中心（datacenterId相同）的不同进程（PID不同）会生成不同的workerId。
    • 不同数据中心（datacenterId不同）的进程即使PID相同，哈希值也不同。

  2. 分布式环境适配：

    • 不依赖IP或MAC地址（适合容器化部署）。
    • PID在单机内唯一，结合datacenterId可扩展至分布式场景。

  3. 限制范围：

    • maxWorkerId 通常是雪花算法中workerId的位数上限（如5位=31），需强制约束。

潜在问题与优化

问题：

  1. PID冲突：
    • 短时间重启进程可能导致PID重复（Linux PID默认循环分配）。
  2. 哈希冲突：
    • 不同字符串可能哈希到相同的值（尽管概率低）。
  3. maxWorkerId过小：
    • 若 maxWorkerId=31，而实际节点数超过32，必然冲突。

优化建议：

示例计算

假设：

  • datacenterId = 2
  • PID = 9876
  • maxWorkerId = 31

步骤：

  1. 拼接字符串："29876"。
  2. 计算哈希："29876".hashCode() → 假设为 456789。
  3. 取低16位：456789 & 0xffff = 0x6FD5（十进制 28629）。
  4. 取模：28629 % 32 = 21。

最终返回 21。

总结

设计目标 实现方式 注意事项
唯一性 结合datacenterId和PID生成哈希 需确保PID不重复
范围限制 哈希取模 maxWorkerId + 1 maxWorkerId需合理设置
容错性 忽略PID获取异常 回退方案可能导致冲突概率升高

这种方法在中小规模分布式系统中足够可靠，但若节点数过多或PID不可控，建议改用更稳定的方案（如ZooKeeper分配唯一ID）。

2. 分配标识位

为什么是分配标识位而不是主动生成标识位

如果每个节点主动生成标识位（如通过MAC地址、IP哈希、PID等），可能因生成逻辑的局限性导致不同节点生成相同的ID。
例如：

  • 两个容器的MAC地址可能相同（虚拟化环境）。
  • 不同机器的IP哈希后可能映射到相同的值（哈希冲突）。
  • 通过集中式配置（如配置文件、数据库、ZooKeeper）显式分配唯一标识，确保全局唯一，或者业务内唯一

标识位如何定义才能不重复？有两种方案：预分配和动态分配

预分配

应用上线前，统计当前服务的节点数，人工去申请标识位。

这种方案，没有代码开发量，在服务节点固定或者项目少可以使用，但是解决不了服务节点动态扩容性问题。

动态分配

通过将标识位存放在 Redis、Zookeeper、MySQL 等中间件，在服务启动的时候去请求标识位，请求后标识位更新为下一个可用的。

通过存放标识位，延伸出一个问题：雪花算法的 ID 是 服务内唯一还是全局唯一。

以 Redis 举例，如果要做服务内唯一，存放标识位的 Redis 节点使用自己项目内的就可以；如果是全局唯一，所有使用雪花算法的应用，要用同一个 Redis 节点。

两者的区别仅是 不同的服务间是否公用 Redis。如果没有全局唯一的需求，最好使 ID 服务内唯一，因为这样可以避免单点问题。

服务的节点数超过 1024，则需要做额外的扩展；可以扩展 10 bit 标识位，或者选择开源分布式 ID 框架。

动态分配实现方案

Redis 存储一个 Hash 结构 Key，包含两个键值对：dataCenterId 和 workerId。

在应用启动时，通过 Lua 脚本去 Redis 获取标识位。dataCenterId 和 workerId 的获取与自增在 Lua 脚本中完成，调用返回后就是可用的标示位。

具体 Lua 脚本逻辑如下：

  1. 第一个服务节点在获取时，Redis 可能是没有 snowflake_work_id_key 这个 Hash 的，应该先判断 Hash 是否存在，不存在初始化 Hash，dataCenterId、workerId 初始化为 0。
  2. 如果 Hash 已存在，判断 dataCenterId、workerId 是否等于最大值 31，满足条件初始化 dataCenterId、workerId 设置为 0 返回。
  3. dataCenterId 和 workerId 的排列组合一共是 1024，在进行分配时，先分配 workerId。
  4. 判断 workerId 是否 != 31，条件成立对 workerId 自增，并返回；如果 workerId = 31，自增 dataCenterId 并将 workerId 设置为 0。

Hutool怎么使用Builder模式创建线程池

默认创建的线程池，构造方法最多有七个参数，核心线程数、最大线程数、阻塞队列、线程存活时间…

日常使用创建线程池时，大家想一下为什么要这么设计？一起来看下源码注释中如何解释此行为。

线程池之所以设置如此之多的构造参数，是因为对这些参数会有一定规则的校验，如果不满足线程池的规则，将不允许创建线程池，通过抛异常的方式终止程序。

终止规则大概有七点，这里列举一下：

  1. 核心线程数不可以小于 0。
  2. 线程存活时间不可以小于 0。
  3. 最大线程数不可以小于等于 0，同时也不可以小于核心线程数。
  4. 阻塞队列、线程工厂、拒绝策略参数均不可为空。

上述七点有两个作用，其一是为了让核心参数满足线程池运行流程，其二是为了保障运行时的稳定性

构造器函数过多以及属性之间存在依赖关系和约束条件。非常适合构建者模式

Hutool Builder 创建线程池

先从 ExecutorBuilder#create 入手

为什么会出现自己创建自己呢

如果你想要对一个类中属性进行约束，前提是不是先应该把属性搞到手。

没错，ExecutorBuilder#create 方法返回自己本身，然后通过 set 方法 把数据填充到创建出来的对象上，最后再进行依赖关系整理和条件约束。

逻辑包括：

  • 依赖注入（如线程工厂、拒绝策略的默认值）。
  • 最终产品构造（调用 ThreadPoolExecutor 构造函数）。

这个时候我们就可以去看一下构建者模式具体的体现

最后 build(ExecutorBuilder builder) 返回的就是我们所需要的线程池对象，这一块大家可以自己跟下源码，学会就可以套用自己写的业务代码。

项目当中什么场景使用了线程池

什么是线程池

线程池是一种基于池化思想管理线程的工具，使用线程池可以减少创建销毁线程的开销，避免线程过多导致系统资源耗尽。在高并发以及大批量的任务处理场景，线程池的使用是必不可少的。

线程过多会带来额外的开销，其中包括创建销毁线程的开销、调度线程的开销等等，同时也降低了计算机的整体性能。线程池维护多个线程，等待监督管理者分配可并发执行的任务。这种做法，一方面避免了处理任务时创建销毁线程开销的代价，另一方面避免了线程数量膨胀导致的过分调度问题，保证了对内核的充分利用。

什么场景下用线程池

线程池在各场景下的核心作用解析

线程池的本质是 复用线程、控制并发、管理资源，不同场景下其核心价值有所不同，以下是逐场景的深度解析：

1. 高并发任务处理（如Web服务器）

核心问题：
每个请求创建一个线程会导致：

  • 线程创建/销毁开销大（Java线程约1MB内存，创建耗时约1ms）。
  • 线程数爆炸引发OOM（如10000并发需10GB内存）。

线程池解决方案：

  • 复用线程：固定数量线程（如200个）循环处理请求，避免频繁创建。
  • 队列缓冲：超出的请求排队，避免瞬时压力击垮系统。
  • 资源可控：通过maxPoolSize限制最大线程数，保护系统。

数据支撑：

  • Tomcat默认线程池（maxThreads=200）可轻松支撑5000+ QPS，而直接new Thread()在1000并发时就会崩溃。

2. 异步任务（如日志记录）

核心问题：
同步写日志会阻塞业务线程，导致响应时间变长。

线程池解决方案：

  • 异步化：日志任务投递到独立线程池，业务线程立即返回。
  • 削峰填谷：突发日志量由队列缓冲，避免影响主流程。

3. 定时任务（如缓存刷新）

核心问题：

  • Timer单线程执行，任务阻塞会导致后续任务延迟。
  • 手动管理多线程定时任务复杂度高。

线程池解决方案：

  • ScheduledThreadPool：
    • 内置时间轮算法，精准控制任务调度。
    • 多线程并行执行，避免任务阻塞。

对比实验：

方案 10个任务耗时 任务阻塞影响
Timer单线程 1000ms 所有任务延迟
ScheduledThreadPool 100ms 仅阻塞任务受影响

4. 批量数据处理（如Excel导入）

核心问题：
单线程处理10万行数据耗时过长（如5分钟）。

线程池优化方案：

  • 分片并行：将数据拆分为100份，线程池并行处理。
  • 动态调节：根据数据量调整线程数（如CPU密集型任务设线程数=CPU核数）。

核心问题：
日志线程占用过多资源，导致支付接口响应变慢。

线程池解决方案：

    • 支付使用FixedThreadPool(50)（高优先级）。
    • 日志使用CachedThreadPool（低优先级）。
  • 拒绝策略：日志队列满时直接丢弃，确保支付业务不受影响。

6. 流量削峰（如秒杀系统）

核心问题：
瞬时1万请求直接压垮数据库。

线程池+队列方案：

  1. 请求先进入LinkedBlockingQueue(5000)。
  2. 线程池（core=100, max=500）按处理能力消费。
  3. 超限请求返回”系统繁忙”。

优势：

  • 数据库每秒只承受500请求（可控）。
  • 90%请求在200ms内响应（队列缓冲）。

7. 避免线程创建开销（如短时RPC调用）

实验对比：

方案 执行10000次耗时 内存占用
每次new Thread() 12秒 10GB
线程池（8线程） 1.5秒 50MB

原因：

  • 线程创建成本：约1ms/次，包括：
    • 分配栈内存（默认1MB/线程）。
    • 系统调用（Linux的pthread_create）。
  • 线程池复用线程，消除创建开销。

线程池的不可替代性

  1. 成本控制：线程是昂贵资源，必须复用。
  2. 稳定性：避免线程数失控导致系统崩溃。
  3. 可观测性：通过ThreadPoolExecutor可监控：
    • 活跃线程数getActiveCount()
    • 队列堆积getQueue().size()

错误案例：
某电商直接new Thread()处理订单，大促时：

  • 创建50万线程 → 内存耗尽 → 全站宕机。
    修复后：
  • 改用线程池（core=200, max=1000, queue=10000）→ 稳定支撑10万订单/分钟。

总结：为什么用线程池？

场景 核心需求 线程池提供的价值
高并发请求 快速响应，避免崩溃 复用线程 + 队列缓冲
异步任务 不阻塞主流程 任务投递后立即返回
定时任务 精准调度，避免阻塞 多线程并行 + 时间轮算法
批量数据处理 加速处理 分片并行 + 资源控制
资源隔离 关键业务不被影响 独立线程池 + 优先级管理
流量削峰 平滑突发流量 队列缓冲 + 拒绝策略
短生命周期任务 避免创建开销 线程复用 + 动态伸缩

终极目标：
以最小资源消耗，提供最高效的并发处理能力。

12306 如何使用线程池

12306 中有个很好的场景，如果一个用户购买多种类型的车座，比如同一订单购买了一等座和二等座，那么就要按照不同的座位逻辑进行选座。

上述这个流程在我们系统里是串行的，也就是说执行完一等座选座后再执行二等座。这个逻辑是可以并行的，可以极大程度上提高咱们的执行效率，提升系统吞吐量。

1. 原始座位分配逻辑

根据上文描述，第一版座位分配场景通过串行的方式分配座位。如果遇到一个用户购买了两到三个座位类型，性能执行稍差。

  • 顺序性：forEach 遍历 seatTypeMap 时，虽然Java不保证 HashMap 的遍历顺序，但实际业务中通常会通过排序或TreeMap确保按固定顺序（如先商务座→一等座→二等座）。
  • 阻塞式处理：每个座位类型的处理必须完成后才会进入下一个类型，前序失败会直接抛异常终止。

2. 代入线程池场景

第二个版本，我们引入线程池去做这个单订单多座位购票场景。

按座位类型拆分任务：每种座位类型的分配作为独立任务

不同座位类型互不影响：

  • 商务座任务卡顿不会阻塞二等座处理
  • 可通过线程池监控单独观察各类型处理情况

3. 线程池执行方案

第二种解决方案看着是挺好的，但是还有一些并行以及性能的问题，如下：

  1. 如果用户下单仅选择单一座位，通过线程池执行会额外增加性能消耗。
  2. 线程池并行的情况下，向 actualResult 添加数据，会存在并发问题。因为 actualResult 是 ArrayList。线程不安全

12306 线程池参数如何设置

1. 有问题的线程池参数

创建一个类似于这种线程池，业务使用会不会有问题？

我们先不说核心线程数和最大线程数是否合理，这个不在本次文章讨论范围内。

从阻塞队列以及拒绝策略的方式进行解析。阻塞队列设置 LinkedBlockingQueue，容量设置为 4096 有什么问题？拒绝策略设置 AbortPolicy 有什么问题？

线程池线程数是有限的，如果大量任务堆积在阻塞队列，这对于业务的执行是有比较大的延迟。

我们期望的结果是，线程池不堆积任务，如果线程池的线程满了就让提交任务的主线程（Web容器执行线程）执行分配座位逻辑。简单来说，就是线程池线程都在执行任务时，我们将并行的逻辑降级为串行。

2. 重构后的线程池参数

为了实现上述业务逻辑，我们通过 Hippo4j 修改创建线程池的阻塞队列和拒绝策略。

将阻塞队列设置为 SynchronousQueue，一个没有容量的阻塞队列，没有容量自然也不会有堆积。

如果线程池满了会触发拒绝策略，如果抛异常就违反了咱们设计的初衷。从 AbortPolicy 修改为 CallerRunsPolicy，代表如果触发拒绝策略，则由提交任务的线程运行该任务。
