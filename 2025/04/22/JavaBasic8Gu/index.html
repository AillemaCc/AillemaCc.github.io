为什么java解释和编译都有

首先在Java经过编译之后生成字节码文件，接下来进入JVM中，就有两个步骤编译和解释。Java源代码首先被编译成字节码，JIT 会把编译过的机器码保存起来,以备下次使用。JVM中一个方法调用计数器，当累计计数大于一定值的时候，就使用JIT进行编译生成机器码文件。否则就是用解释器进行解释执行，然后字节码也是经过解释器进行解释运行的。所以Java既是编译型也是解释性语言，默认采用的是解释器和编译器混合的模式。

自动类型转换

强制类型转换

数值之间的转换：将整型转换为字符型、将字符型转换为整型等。这些转换方式可以通过类型的包装类来实现。例如Character类、Integer类等提供了相应的转换方法。

Java为什么要有Integer？

Integer对应是int类型的包装类，可以把数据跟处理这些数据的方法结合在一起。比如Integer就有parseInt()等方法来专门处理int型相关的数据。另一个非常重要的原因就是在Java中绝大部分方法或类都是用来处理类类型对象的

在泛型当中，泛型只能使用引用类型，必须使用包装类。在转换当中，基本类型和引用类型不能直接进行转换，必须使用包装类来实现。例如，将一个int类型的值转换为String类型，必须首先将其转换为Integer类型，再转换成String。而在集合当中，只能存储对象，而不能存储基本数据类型。

多态体现在哪几个方面

多态是指允许不同类的对象对同一消息作出响应。

比如方法的重载，同一类中可以有多个同名方法。方法的重写，子类提供对父类同名方法的具体实现。

同样的，对于接口来说，可以有很多不同的实现类。向上转型向下转型也体现多态的部分。

抽象类和普通类区别

  • 实例化：普通类可以直接实例化对象，而抽象类不能被实例化，只能被继承
  • 方法实现：普通类中的方法可以有具体的实现，而抽象类中的方法可以有实现也可以没有实现。
  • 继承：一个类可以继承一个普通类，而且可以继承多个接口；而一个类只能继承一个抽象类，但可以同时实现多个接口。
  • 实现限制：普通类可以被其他类继承和使用，而抽象类一般用于作为基类

抽象类和接口的区别

抽象类用来描述类的共同特性和行为，可以有成员变量，构造方法或者具体方法。但是接口用来定义行为规范，只能有常量和抽象方法。

在具体区别上，首先是实现方式不同，接口用implements实现，继承抽象类使用extends。其次在方法方式上，接口只有定义，不能有方法的实现。抽象类可以有定义和实现。对于访问修饰符，接口的成员变量默认为public static final，必须赋初值，不能被修改。所有的方法都是public、abstract的

抽象类可以被实例化吗

抽象类本身不能被实例化。这意味着不能使用new关键字直接创建一个抽象类的对象。抽象类的存在主要是为了被继承。包含一个多个抽象方法的抽象类，可以在它的子类当中实现这些抽象方法。子类当然可以被实例化。所以说这也是一种实例化。

final的作用

从上到下修饰类，修饰方法，修饰变量。

修饰类，表示这个类不能被继承。修饰方法，被final修饰的方法不能被子类重写。修饰变量，被final修饰的基本数据类型变量一旦被赋值就不能被改变。被final修饰的引用数据类型就不能再指向其他变量了。但是这个对象本身的内容还是可以改变的。

什么是泛型

泛型允许类接口和方法在定义的时候，使用一个或者多个类型参数。这些类型参数可以在使用时才制定具体的类型。泛型的主要目的是在编译的时候提供更强的类型检查，并且在编译之后可以保留类型信息，避免在运行的时候出现类型转换异常。

泛型中的类型在使用时指定就可以了，不需要强制类型转换。

list当中的元素都是object类型，所以在取出元素的时候，需要人为强转成具体的目标类型。假如提供泛型，就可以提供类型的约束。

Java并发当中的线程安全

原子性，可见性，有序性。可见性指一个线程对主内存的修改，必须可以及时的被其他线程看到。在java当中使用synchronized和volatile这两个关键字确保可见性。另外一点事有序性，因为一个线程在观察其他线程中的指令执行顺序，因为指令重排序，所以观察的结果会杂乱无序。java当中使用happens-before原则来确保有序性

偏向锁升级轻量级锁的过程

偏向锁升级到轻量级锁的过程通常发生在有多个线程尝试获取同一个锁，而该锁当前正处于偏向锁状态时。以下是偏向锁升级为轻量级锁的详细过程：

  1. 初始偏向：当一个线程第一次获取锁时，如果目标对象没有被其他线程锁定（即处于无锁或偏向锁状态），JVM会在对象头中标记该对象为偏向锁，并记录下持有该锁的线程ID。

  2. 偏向撤销：当另一个线程尝试获取已经被某个线程偏向的锁时，需要先检查偏向锁是否仍然有效：

    • 如果原先持有锁的线程已经结束或者不再持有该锁，那么偏向锁可以被撤销，对象恢复到无锁状态。
    • 如果原先持有锁的线程还在使用该锁，此时就会触发偏向锁的撤销操作。

  3. 偏向锁撤销：偏向锁的撤销可能会导致一次全局的安全点（Safe Point），这会暂停所有Java线程（Stop-The-World, STW）。撤销过程中，JVM会将对象的状态从偏向锁转换为无锁状态，然后重新竞争锁。

  4. 轻量级锁的加锁：在偏向锁撤销后，如果有多个线程竞争同一把锁，JVM会尝试使用CAS（Compare And Swap）操作来为这些线程分配轻量级锁。每个竞争的线程都会在自己的栈帧中创建一个锁记录（Lock Record），并尝试通过CAS操作将对象头中的Mark Word更新为自己锁记录的指针。

  5. 自旋等待：如果CAS操作失败，说明已经有其他线程成功获得了轻量级锁，当前线程会进入一种称为“自旋”的状态，在一段时间内循环尝试获取锁，而不是立即进入阻塞状态。

  6. 升级重量级锁：如果自旋最终未能成功获取锁，且达到一定的自旋次数限制或检测到自旋期间没有取得进展，锁将会膨胀为重量级锁，此时未获得锁的线程会被操作系统挂起，直到持有锁的线程释放锁为止。

这个过程是JVM为了提高并发性能而设计的一种优化策略，通过逐步升级锁的方式，尽量减少不必要的上下文切换和线程阻塞，以提高系统的整体效率。需要注意的是，一旦锁升级为重量级锁之后，就不会再降级回轻量级或偏向锁。

封装 继承 多态

抽象类和接口的区别

首先，抽象类是描述类的共同特性和行为，接口则是对方法的规范。他们两个继承和实现的关键字不同，对于他们的变量来说，抽象类当中可以有变量，包括实例变量和静态变量。而接口当中只能有常量。对于他们的方法来说，抽象类当中的方法可以有具体的实现，当然也可以没有。但是接口当中的方法不能有具体的实现。java不支持多继承，所以一个普通类只能继承一个抽象类，但是支持多实现，一个普通类可以实现多个接口。

对象头

在jvm当中，每个对象的内存布局主要由两部分组成

  • Mark Word；用于存储对象运行时的数据，包括GC分代，哈希码，锁状态等等
  • Klass Pointer；指向对象类型的元数据

Mark Word是实现Synchronized的锁膨胀的关键

在四种锁膨胀状态下，他会逐渐存储不同的信息

  • 无锁；存储修饰的对象的哈希码和分代信息
  • 偏向锁，先备份当前对象的无锁状态，然后保存能够获取该锁的线程ID
  • 轻量级锁，备份当前对象的无锁状态，通过cas操作尝试将对象头中的Mark Word更新为指向自己的指针
  • 重量级锁，轻量级锁竞争失败就膨胀为重量级锁

偏向锁到轻量锁

你的理解基本正确，但偏向锁和轻量级锁的细节稍有不同。我来帮你更精确地梳理它们的加锁过程，并指出关键区别：

1. 偏向锁（Biased Locking）

核心特点

  • 适用场景：只有一个线程反复访问同步块。
  • 优化目标：完全避免同步开销（连 CAS 都不需要）。
  • 对象头变化：直接存储线程ID，而非锁记录指针。

加锁流程

  1. 初始状态（无锁）
  2. 首次加锁（CAS 设置偏向线程ID）
    • 检查biased_flag是否为 1（可偏向）且线程ID为空。
    • CAS 将 Mark Word 替换为当前线程ID（并保留分代年龄等元数据）：
    • 不需要备份原 Mark Word（因为未覆盖元数据，线程ID直接存入空闲位）。
  3. 后续加锁
    • 检查 Mark Word 中的线程ID是否为本线程：
      • 是：直接进入同步块（无任何原子操作）。
      • 否：撤销偏向锁，可能升级为轻量级锁。

关键区别

  • 不依赖 Lock Record，线程ID直接嵌入 Mark Word。
  • 无 CAS 竞争（只有首次设置线程ID需要 CAS）。
  • 不备份原 Mark Word（因为未破坏元数据）。

2. 轻量级锁（Thin Lock）

核心特点

  • 适用场景：多个线程交替访问同步块，但无激烈竞争。
  • 优化目标：用 CAS 代替操作系统互斥锁。
  • 对象头变化：存储指向线程栈中 Lock Record 的指针。

加锁流程

  1. 初始状态（无锁或偏向锁已撤销）
  2. 创建 Lock Record
    • JVM 在当前线程栈帧中分配 Lock Record。
    • 备份原 Mark Word 到 Lock Record 的 Displaced Mark Word 字段（必须备份，因为 CAS 会覆盖整个 Mark Word）。
  3. CAS 竞争锁
    • 尝试将 Mark Word 更新为指向 Lock Record 的指针，并标记为轻量级锁（00）：
    • 成功：线程获得锁。
    • 失败：说明其他线程已竞争，触发锁膨胀（升级为重量级锁）。

关键区别

  • 必须备份原 Mark Word（因为 CAS 会完全覆盖它）。
  • 依赖 Lock Record 存储原状态，解锁时需恢复。
  • 通过 CAS 竞争（而偏向锁仅在首次设置线程ID时用 CAS）。

3. 你的陈述 vs 实际情况

你的说法 修正说明
偏向锁：备份无锁状态 ❌ 偏向锁不备份 Mark Word，而是直接写入线程ID（保留元数据）。
偏向锁：保存线程ID ✅ 正确。线程ID直接存入 Mark Word 的高位。
轻量级锁：备份无锁状态 ✅ 正确。原 Mark Word 存入 Lock Record 的 Displaced Mark Word。
轻量级锁：CAS 更新为指向自己的指针 ✅ 正确。对象头改为指向线程栈中的 Lock Record。

4. 为什么设计这样的差异？

  1. 偏向锁：

    • 针对单线程重复加锁的场景，连 CAS 都想省去。
    • 线程ID直接嵌入 Mark Word，无需额外存储。

  2. 轻量级锁：

    • 针对短时间多线程交替访问，需通过 CAS 实现竞争。
    • 备份原 Mark Word 是必须的，因为 CAS 会破坏原有数据（如哈希码）。

总结

  • 偏向锁：不备份，直接存线程ID，适合单线程场景。
  • 轻量级锁：备份原数据，CAS 竞争锁记录指针，适合低竞争场景。
  • 你的理解大部分正确，只需修正偏向锁的备份问题。

这样设计是为了在不同竞争强度下，以最小开销实现线程安全。

轻量级锁到底是什么个过程

竞争过程的出现，会让偏向锁膨胀成轻量级锁。主要的阐述点就是竞争的流程。一个线程想要获取锁，首先在自己的线程内部创建一个锁记录，用来备份偏向锁的对象头。然后线程使用cas操作尝试修改对象头的指针指向，尝试更改为指向锁的记录
