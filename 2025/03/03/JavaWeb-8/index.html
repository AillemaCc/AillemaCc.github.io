预编译sql

预编译SQL有两个优势：

  1. 性能更高
  2. 更安全(防止SQL注入)

性能更高：预编译SQL，编译一次之后会将编译后的SQL语句缓存起来，后面再次执行这条语句时，不会再次编译。（只是输入的参数不同）

更安全(防止SQL注入)：将敏感字进行转义，保障SQL的安全性。

参数占位符

在Mybatis中提供的参数占位符有两种：${…} 、#{…}

  • #{…}

    • 执行SQL时，会将#{…}替换为?，生成预编译SQL，会自动设置参数值
    • 使用时机：参数传递，都使用#{…}

  • ${…}

    • 拼接SQL。直接将参数拼接在SQL语句中，存在SQL注入问题
    • 使用时机：如果对表名、列表进行动态设置时使用

注意事项：在项目开发中，建议使用#{…}，生成预编译SQL，防止SQL注入安全。

外卖菜品与套餐模块的表结构，菜品与套餐是多对多的关系，一个套餐对应多个菜品。既然是多对多的关系，是不是有一张套餐菜品中间表来维护它们之间的关系。

在添加套餐的时候，我们需要在界面当中来录入套餐的基本信息，还需要来录入套餐与菜品的关联信息。这些信息录入完毕之后，我们一点保存，就需要将套餐的信息以及套餐与菜品的关联信息都需要保存到数据库当中。其实具体的过程包括两步，首先第一步先需要将套餐的基本信息保存了，接下来第二步再来保存套餐与菜品的关联信息。套餐与菜品的关联信息就是往中间表当中来插入数据，来维护它们之间的关系。而中间表当中有两个外键字段，一个是菜品的ID，就是当前菜品的ID，还有一个就是套餐的ID，而这个套餐的 ID 指的就是此次我所添加的套餐的ID，所以我们在第一步保存完套餐的基本信息之后，就需要将套餐的主键值返回来供第二步进行使用。这个时候就需要用到主键返回功能。

那要如何实现在插入数据之后返回所插入行的主键值呢？

  • 默认情况下，执行插入操作时，是不会主键值返回的。如果我们想要拿到主键值，需要在Mapper接口中的方法上添加一个Options注解，并在注解中指定属性useGeneratedKeys=true和keyProperty=”实体类属性名”

在使用MyBatis进行数据库操作时，useGeneratedKeys和keyProperty是两个非常重要的属性，主要用于处理插入数据后获取自动生成的主键值。下面详细解释这两个属性的作用以及如何正确使用它们。

useGeneratedKeys

  • 作用: 当设置为true时，MyBatis会使用JDBC的getGeneratedKeys方法来获取由数据库自动生成的主键值，并将其赋值给指定的属性。
  • 默认值: false
  • 适用场景: 主要用于支持自增主键的数据库（如MySQL、PostgreSQL等）。

keyProperty

  • 作用: 指定一个实体类的属性名，MyBatis将从数据库获取的自动生成的主键值赋值给这个属性。
  • 默认值: 无
  • 适用场景: 需要在插入记录后立即获取其主键值的情况。

注意事项

  1. 数据库支持：确保你的数据库支持自增主键，并且表结构中定义了自增字段（如MySQL中的AUTO_INCREMENT）。
  2. 字段类型匹配：确保keyProperty指向的属性类型与数据库返回的主键类型相匹配（例如，如果主键是INT类型，则实体类中的对应属性也应该是Integer或int类型）。
  3. 多列主键：如果表使用的是复合主键（即多个列组合成唯一标识），则useGeneratedKeys和keyProperty可能不适用，需要考虑其他方式获取主键值。

通过正确配置和使用useGeneratedKeys和keyProperty，你可以方便地在插入记录后获取到数据库生成的主键值，并进一步利用这些值进行后续的操作。

反射的基本概念

反射（Reflection） 是Java语言的一个特性，它允许程序在运行时检查和操作类、方法、字段等结构信息。具体来说，反射可以：

  • 获取类的构造函数、方法、字段等信息。
  • 调用方法或访问字段，即使这些方法或字段是私有的。
  • 创建对象实例。
  • 获取和设置对象属性值。

在MyBatis中，反射主要用于将方法参数绑定到SQL语句中的占位符，并从查询结果中映射数据到Java对象。

结合 -parameters 选项的反射机制

当启用 -parameters 编译选项时，编译器会在生成的字节码文件中保留方法参数名称。这样，MyBatis可以通过反射机制获取这些参数名称，并将其与SQL语句中的占位符匹配起来。

步骤解析

  1. 方法签名分析：

    • MyBatis首先需要知道你的方法签名，即 list(String name, Short gender, LocalDate begin, LocalDate end)。
    • 由于启用了 -parameters 选项，编译器在生成的字节码中保留了这些参数名称（name, gender, begin, end）。

  2. 反射获取参数名称：

    • 在运行时，MyBatis通过Java反射API（如 Method.getParameterNames() 或其他反射工具）获取方法参数名称。
    • 这些参数名称被存储在一个映射表中，用于后续的参数绑定。

  3. 参数绑定：

    • 当执行查询时，MyBatis会解析SQL语句中的占位符（如 #{name}, #{sex}, #{begin}, #{end}）。
    • MyBatis根据这些占位符名称查找对应的参数名称（例如，#{name} 对应于方法参数 name）。
    • 然后，MyBatis将实际传入的方法参数值绑定到相应的占位符上。

  4. 执行SQL查询：

    • 最终，MyBatis构建并执行带有绑定参数的SQL查询。
    • 查询结果会被映射回Java对象（如 Emp 类的实例）。

反射的作用

在上述过程中，反射的主要作用体现在以下几个方面：

  1. 获取方法参数名称：

    • 启用 -parameters 编译选项后，编译器保留了方法参数名称。MyBatis通过反射API获取这些名称，并用于参数绑定。

  2. 动态绑定参数值：

    • MyBatis利用反射机制动态地将方法参数值绑定到SQL语句中的占位符，确保查询能够正确执行。

  3. 类型转换和映射：

    • 反射还可以帮助MyBatis进行类型转换和结果映射。例如，将数据库查询结果转换为Java对象的属性值。

总结

在启用了 -parameters 编译选项的情况下，MyBatis通过反射机制获取方法参数名称，并将这些名称与SQL语句中的占位符匹配起来，从而实现参数的动态绑定。反射在这个过程中起到了关键作用，使得我们可以直接在SQL语句中使用形参名作为占位符，而不需要显式地使用 @Param 注解。这不仅简化了代码编写，还提高了开发效率。

多出了一个逗号，哪来的

在你的 <update> 语句中，确实可能会出现多余的逗号问题。具体来说，当多个 <if> 条件中的某一个不满足时（即对应的字段值为 null），生成的 SQL 语句中可能会多出一个逗号，导致语法错误。

为什么会出现多余的逗号？

让我们详细分析一下你提供的 MyBatis 动态 SQL 片段：

分析

  1. 每个 <if> 标签后面都有一个逗号：

    • 每个 <if> 标签后面都跟了一个逗号 ,，这意味着如果某个条件为 true，则该条件对应的列会被更新，并且会在 SQL 语句中添加一个逗号。

  2. 最后一个 <if> 标签没有逗号：

    • 最后一个 <if> 标签（update_time）没有加逗号，这是为了避免在最后一个条件为 true 时多出一个逗号。

  3. 问题出现在中间部分：

    • 如果某些条件为 false（即对应的字段值为 null），那么这些条件不会被包含在生成的 SQL 语句中，但前面的条件仍然会保留它们的逗号。
    • 这样会导致在生成的 SQL 语句中，最后一个有效的列和 where 子句之间可能会有一个多余的逗号。

结论

问题的根本原因在于：

  • 在动态 SQL 中，每个 <if> 标签后面都加上了逗号，但在某些情况下（如某些条件为 false），这些逗号会保留在最终生成的 SQL 语句中，导致语法错误（例如，最后一个有效列与 where 子句之间的多余逗号）。

因此，即使你在最后一个 <if> 标签中没有加逗号，其他条件为 false 的情况下仍然可能产生多余的逗号。这就是为什么会出现多余逗号的原因。

以上问题的解决方案：使用<set>标签代替SQL语句中的set关键字

  • <set>：动态的在SQL语句中插入set关键字，并会删掉额外的逗号。（用于update语句中）
